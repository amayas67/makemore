1. The embedding pipeline: from indices to a flat vector

At the start, a context is nothing more than a sequence of character indices.

Context:

[3, 4, 2]

These numbers are just identifiers — they carry no meaning by themselves.

⬇

Step 1 — the embedding matrix C

Each row of C holds the vector associated with one character.

C[3] = [ 0.7, -1.1]
C[4] = [-0.2,  0.5]
C[2] = [ 1.8,  0.3]
⬇

Step 2 — C[X]: the lookup

PyTorch replaces every index with its corresponding vector.

Before:

[3, 4, 2]

After:

[
    C[3],
    C[4],
    C[2]
]

=

[
    [ 0.7, -1.1],
    [-0.2,  0.5],
    [ 1.8,  0.3]
]

The context is no longer 3 numbers — it is now 3 vectors.

⬇

Step 3 — flattening

[
    [ 0.7, -1.1],
    [-0.2,  0.5],
    [ 1.8,  0.3]
]

↓

[0.7, -1.1, -0.2, 0.5, 1.8, 0.3]

This single long vector is what gets fed into the neural network.

2. From logits to a loss

After the output layer, each context has 27 logits — one raw score per possible next character.

logits
 ↓
exp()
 ↓
divide by the row sum
 ↓
probabilities

The column with the highest probability is the character the model would predict.

During training, we don't use that top probability directly. We only pick out the probability the model assigned to the true next character:

probs[torch.arange(len(X)), Y]

torch.arange(len(X)) walks through every row (every context), and Y tells us, for each row, which column holds the true character.

probs
 ↓
probability of the true character
 ↓
log()
 ↓
mean
 ↓
negative sign
 ↓
Loss

The probability of the whole dataset is the product of the probabilities of every true character. Taking the logarithm turns that product into a sum, and the negative sign turns it into something we minimize.

loss = -probs[torch.arange(len(X)), Y].log().mean()

PyTorch does all of this automatically with:

loss = F.cross_entropy(logits, Y)
3. Why use F.cross_entropy()?

We could compute the loss by hand:

counts = logits.exp()
probs = counts / counts.sum(1, keepdim=True)
loss = -probs[torch.arange(len(X)), Y].log().mean()

This is great for understanding what's going on, but it can run into a numerical stability problem.

Example: when logits get too big or too small

Suppose one of the logits is very large:

logits = [1000, 2, -3]

Computing directly:

exp(1000)

The result is so large it overflows what the computer can represent as a floating-point number. You get inf, and every probability downstream becomes garbage.

On the other end, a very negative logit like -1000 gives:

exp(-1000) ≈ 0

which can lead to division by zero or log(0), which is -inf.

F.cross_entropy() avoids all of this. Instead of naively computing exp() then log(), PyTorch uses a mathematically equivalent but far more stable formulation called LogSoftmax. The math is identical — the computation is just numerically safe, even when logits become extreme during training.

Why subtract the max logit?

The trick behind F.cross_entropy() is that before exponentiating, PyTorch subtracts the largest logit in each row from every logit in that row.

logits = [1000, 998, 995]

max(logits) = 1000

↓

[0, -2, -5]

Then:

exp(0)  = 1
exp(-2) ≈ 0.135
exp(-5) ≈ 0.0067

Nothing is gigantic anymore, so the computation stays stable.

Why doesn't this change the probabilities?

Softmax is:

       exp(logit_i)
-------------------------
    Σ exp(logit_j)

If we subtract the same constant M from every logit:

     exp(logit_i - M)
-----------------------------
   Σ exp(logit_j - M)

and since exp(logit_i - M) = exp(logit_i) × exp(-M), the factor exp(-M) appears in both the numerator and the denominator — so it cancels out completely.

Conclusion: the probabilities are exactly the same either way. Only the intermediate numbers become much smaller, which is what avoids overflow (inf) and log(0) issues.
Other advantages of F.cross_entropy()
  • Faster forward pass: PyTorch fuses several operations (log_softmax + the loss computation) into one optimized implementation — fewer operations to run.
  • Faster backward pass: during loss.backward(), PyTorch computes the cross-entropy gradient directly, instead of separately backpropagating through every exp, division, and log we wrote by hand.
  • Less memory: the hand-written version has to keep several intermediate tensors around (counts, probs, …) for the backward pass. Fused F.cross_entropy() stores far fewer intermediate results.
In short
  • ✔ Same mathematical result
  • ✔ Numerically stable
  • ✔ Faster forward pass
  • ✔ Faster backward pass
  • ✔ Less memory used
4. Training dynamics: is C re-randomized every epoch?

No. C is initialized randomly exactly once, before training starts. After that, it is never recreated — every epoch only nudges it a little from where it already was:

C_new = C_old - learning_rate * gradient
This is the same update rule used for every other parameter — W1, b1, W2, b2. The only thing special about C is that its gradient doesn't come directly from the loss: it has to flow backward through the whole network first — logits → h → embd → embeddings → C[X] → C — before it reaches C.

Concretely, over training:

  • Epoch 0: C is random, the embeddings mean nothing yet, the loss is high.
  • Epoch 1: C shifts slightly, in whatever direction would have reduced last epoch's error.
  • … and so on, until the embeddings capture something useful — for instance, vowels ending up close to each other in the 2D space, because they tend to play a similar role in predicting the next character.
One more detail: C has exactly one row per character, not one per position or context. If 'a' shows up in 1,000 different contexts across the dataset, every one of those lookups points to the same row of C. Each occurrence contributes to the gradient of that single row, so the final embedding for 'a' is a compromise — something that works reasonably well across every context where 'a' appears, not a vector fine-tuned for one specific case.
