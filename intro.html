Part 1 — Intuitive Summary

In [01]:# What is our goal?

We want to teach a program to invent names that look like the ones in the dataset.

In [02]:# What does the dataset contain?

About 32,000 names. These serve as our examples.

In [03]:# Why create the N matrix?

To count how many times each pair of letters (bigram) appears.

For example:

  • e → m
  • m → a
  • a → .

The more often a pair appears, the higher its counter.

Out[03]:

The N matrix does NOT contain probabilities. It only contains counts.

In [04]:# What does the P matrix do next?

We transform the counts into probabilities.

After this step, each row answers the question:

"If I am on this letter, what is the probability of choosing each of the following letters?"

In [05]:# Does the first model choose letters completely at random?

No.

The choice remains random, but it is guided by the dataset's probabilities.

If "em" appears very often, the model will choose "m" after "e" much more often than "x".

In [06]:# So why are the generated names still bad?

Because the model only looks at one letter at a time.

It doesn't know the whole beginning of the name.

It only knows:

"I am on the letter e. What is the next letter?"

This is too limited information.

In [07]:# How do we know if our model is good?

We ask it:

Out[07]:

"If you had to rewrite the entire dataset, letter by letter, how confident would you be?"

To answer, it calculates the probability of each transition and then combines them all.

Multiplying tens of thousands of small probabilities gives a result far too small to be represented by the computer (this is called numerical underflow). Logarithms turn this product into a sum, which stays numerically stable.

In [08]:# Why do we talk about Negative Log-Likelihood (NLL)?

Because it is more convenient to minimize a number than to maximize a probability.

Out[08]:

Low NLL = the model assigns a high probability to the dataset.

High NLL = the model assigns a low probability to the dataset.

In [09]:# What happens if a pair of letters never appears?

The probability is 0. The log of 0 is mathematically defined, but it tends toward negative infinity — an unusable value for our calculation. So we add a "ghost" (+1) to every cell in the matrix to avoid ever hitting this case.

Out[09]:

This is called Model Smoothing. It ensures the model never encounters a strictly zero probability.

In [10]:# What will the neural network do?

We are not going to give it rules.

We will only tell it:

Out[10]:

"Try to rewrite the dataset, letter by letter, with as few errors as possible."

With each error, its parameters will be slightly modified.

Gradually, it will discover for itself which letter sequences are the most plausible.

In [11]:# Is the goal to memorize the dataset?

No.

The goal is to understand the regularities of the names.

We hope that by learning to rewrite the dataset examples well, the model will also be able to invent new, credible names.

Part 1b — From Counting to Neurons

In [12]:# How do we turn a letter into something a neural network can multiply?

We use one-hot encoding: a vector full of zeros, with a single 1 at the position of that letter.

Example, if "a" is index 1 out of 27 possible characters:

# "a" is index 1 out of 27 possible characters
[0, 1, 0, 0, 0, ..., 0]   # 27 numbers, only one of them is 1
In [13]:# Does PyTorch do something special when it sees a one-hot vector?

No. It performs a completely normal matrix multiplication, with no shortcut.

It's just that math itself makes every term with a 0 disappear. What remains is exactly one row of the weight matrix W - the row that matches the position of the 1.

one_hot(letter) @ W   # same as W[letter], written as a matmul so it stays trainable
In [14]:# If W has one row per letter, are these rows independent of each other?

Yes. Since only one row gets "activated" at a time, only that row can receive a gradient during that pass.

Adjusting the row for "a" never touches the row for "b" - they never appear together in the same one-hot vector.

In [15]:# Can a weight already have a gradient before we even feed it an input?

No. A gradient is not something a weight carries around permanently - it is recomputed from scratch at every training step, and only after three things happen, in this order:

Out[15]:

1. Forward pass (input goes through the network) →
2. Loss is computed →
3. Backward pass computes the gradient

There is no such thing as "the gradient" sitting there waiting before an input arrives. It is always a consequence of some specific input having just passed through.

In [16]:# So which rows of W actually get updated at each step?

Only the rows of the letters that were actually used as input in that batch.

If a letter never appears as the first letter of any bigram in the whole dataset, its row in W never receives a gradient, and it never moves from its random starting point.

In [17]:# Why does the output layer need 27 neurons instead of just 1?

Because the task is not "output one number" - it's "tell me how likely each of the 27 possible next letters is, all at once".

One neuron can only output one number, so it can encode one dimension of information (roughly: "high or low"). It cannot separately represent 27 independent probabilities at the same time.

Out[17]:

Same idea as color: one number can only give you a grayscale level. To get real color, you need three independent numbers (red, green, blue). To get a probability for each of 27 letters, you need 27 independent numbers.

In [18]:# What does tanh do, and why isn't it used on the final output?

tanh squashes a number into the range -1 to 1. It's used inside hidden layers to add non-linearity - without it, stacking layers would collapse into one big linear operation.

But tanh outputs don't sum to 1, so they cannot be read as a probability distribution. For the final layer, we use softmax instead, which turns the 27 raw scores into 27 probabilities that always sum to 1.

In [19]:# Does each output neuron "know" in advance which letter it's predicting?

No, and this is easy to misunderstand. At the start, weights are random garbage - the neuron at position 0 knows nothing about the character ".".

It only becomes "the detector for . " because of a fixed convention we chose ourselves (position 0 = the character "."), combined with training repeatedly pushing that position's score up whenever the true next character is "." and down otherwise.

Out[19]:

The meaning of a neuron isn't built in - it's sculpted by training, guided entirely by the labeling convention we picked when building the vocabulary.

In [20]:# Does the probability of predicting "." depend only on the weights feeding that one position?

No. Softmax's denominator adds up the scores from all 27 neurons. So the probability assigned to "." can shift even if the weights feeding position 0 never changed - simply because the weights feeding some other position did.

Every output position is coupled to every other output position through this shared normalization.

In [21]:# Could improving one prediction accidentally make another one worse?

Yes - and it's unavoidable whenever two training examples share the same input letter but expect different next letters.

Example: "an" and "al" both start from "a". Pushing P(n | a) up and pushing P(l | a) up both pull on the exact same 27 numbers, through the exact same softmax. Since all 27 probabilities must sum to 1, raising one very slightly lowers the others.

Out[21]:

This isn't a bug. The tug-of-war settles at exactly the point that reflects how often "an" and "al" each appear in the dataset - which is exactly what the counting-based model (N and P matrices) computed directly. Gradient descent rediscovers the same answer, one small step at a time.

Part 1c — Confirming Understanding: Row Selection & Gradient Flow

In [22]:# By convention, what does output neuron index j represent, and what does probs[i][j] mean?

By convention, output neuron 0 corresponds to the score of ".", neuron 1 to "a", neuron 2 to "b", and so on. Since xenc is a one-hot vector, the multiplication xenc @ W selects the row of W matching the input character - that row contains the 27 scores produced by the output neurons.

After applying exp and normalizing (softmax), those scores become probabilities.

probs[i][j]  # probability of predicting character j, given that the input was xs[i]
In [23]:# Concretely, which weights actually receive a gradient for a given example?

Only the row of W that was selected for that example. Since logits[i] only depends on row k = xs[i] of W, the backward pass can only push a nonzero gradient into that one row for that example - every other row's gradient contribution from this example is exactly 0.

xs[i] = 5
W.grad[5]      # nonzero — updated from this example
W.grad[0:5]    # zero — untouched
W.grad[6:]     # zero — untouched
Out[23]:

Over a full batch, several rows can end up with nonzero gradients at once - one per distinct input character in the batch - but each row's gradient only ever comes from the examples where that specific character was the input. This is the same row-independence property already described above, just seen from the gradient side instead of the forward pass side.

In [24]:# In probs[i][j], does i refer to the character value or to its position in the batch?

Its position in the batch - not the character value. If xs = [0, 5, 13, 13, 1], then probs has shape (5, 27): 5 because there are 5 examples in this batch, not because there are 5 possible characters.

Row i of probs is the result for the i-th example in the list, whatever character that example's input happens to be. The row of W that gets selected during the forward pass depends on the input's value (xenc[i] @ W picks out W[xs[i]]) - but that result still gets written to row i of probs, i.e. its position in the batch.

xs = [0, 5, 13, 13, 1]
probs.shape       # (5, 27)

probs[0, 5]    # example 0, input xs[0] = 0 → row index (0) happens to match the character value
probs[1, 13]   # example 1, input xs[1] = 5 → uses W[5] internally, output still lands in probs[1]
Out[24]:

There is no "P[5][13]" here - that would require a (27, 27) matrix indexed by character value, like the N/P matrices from Part 1a. This probs tensor is (batch size, 27), indexed by position in the batch.

Check it yourself: F.one_hot(torch.tensor([5]), 27).float() @ W, then softmax, gives exactly probs[1] - not a coincidence, just a consequence of xenc @ W depending only on the character's value, never on its position in the batch.

In [25]:# Is the bigram neural net basically just a fancy lookup table?

Yes - and this intuition is correct. The bigram model (W of size 27x27) is mathematically equivalent to the N/P counting matrices from Part 1a: same information, just encoded differently (trained weights instead of raw counts). One character in, one row selected, one set of scores out - a correspondence table, dressed up as a matrix multiplication so it can be trained with gradients instead of filled in by hand.

In [26]:# Why doesn't a lookup table scale to longer context (e.g. the last 10 characters instead of just 1)?

A raw lookup table needs one row per possible context. With an alphabet of 27 characters and a context of length k, that's 27^k rows: 27 for k=1, 19 683 for k=3, roughly 2x10^14 for k=10.

Out[26]:

Two separate problems, not one:

1. Memory - a table that size can't realistically be stored.

2. Data sparsity (the deeper problem) - even if it could be stored, a dataset of ~32 000 names would leave almost every one of those 27^10 contexts with a count of exactly 0. There's nothing to learn from an empty cell - this isn't a storage limit, it's a "not enough data to ever fill the table" limit.

What a real neural network (with embeddings, added later in the series) actually buys isn't just a smaller table - it's generalization. Instead of memorizing each context as an isolated cell, the network learns a shared representation where similar contexts (e.g. "cha" and "cho") produce similar predictions, even if one of them was never seen during training. That's the real reason this architecture matters - not that the table gets compressed, but that the model no longer needs to have seen every combination to predict on it.

In [27]:# Why does the trained network's loss stall at roughly the same value as the counting model's loss?

Because both models converge to the same underlying information: the empirical frequency of the next character given the previous one.

The counting model (P) directly encodes how often each next character follows a given character. The neural network, through gradient descent, adjusts the row of W for a given input every time that input appears in the dataset - so a row gets pulled toward the optimal solution in exact proportion to how often its character shows up. The neural net's real advantage isn't beating this bigram baseline - it's that this same architecture extends naturally to longer context (multiple preceding characters), unlike a raw counting table (see Q25/Q26).

Out[27]:

Minimizing cross-entropy loss by gradient descent, with no regularization term, converges mathematically to the raw maximum-likelihood estimate - the exact dataset frequencies, with no smoothing. The P matrix, on the other hand, has +1 Laplace smoothing baked in. Both converge to the same underlying distribution, but the counting model carries a small bias from that +1 - which is exactly why the two losses land close but not identical. (Adding an L2 regularization term on W would play a role equivalent to smoothing here: the stronger the regularization, the more it pulls predictions toward uniform - like adding more fake counts.)
